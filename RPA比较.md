## 一、 维度一：流程文件的 LLM 识别与读写能力
**结论**：**目前的 RPA 工具在“流程文件”层面上对 LLM 都不够友好，都是“黑盒”或“噪音”。**
### 1. 影刀 RPA (ShadowBot)
*   **文件格式**：**私有二进制/加密格式 / 数据库存储**。
*   **LLM 读写能力**：⭐ (低 - 文件级) / ⭐⭐⭐⭐ (代码块级)
*   **深度解析**：
    *   **文件不可读**：影刀的项目文件是高度封装的，您无法用文本编辑器打开，也无法直接丢给 LLM 让其“通读”整个流程逻辑。LLM 无法理解影刀的`.shadowbot`文件结构。
    *   **曲线救国**：虽然**流程文件**不可读，但影刀不仅支持 Python，而且鼓励使用 Python。您可以让 LLM 生成纯 Python 代码，然后复制粘贴到影刀的“运行 Python 模块”中。**这是“逻辑层面的 LLM 友好”，而非“文件层面的 LLM 友好”。**
### 2. UiPath
*   **文件格式**：**.xaml** (基于 XML 的文本文件)。
*   **LLM 读写能力**：⭐⭐ (中下 - 噪音太大)
*   **深度解析**：
    *   **理论可读，实际难改**：.xaml 是明文，LLM 能读懂 XML 标签。但是，UiPath 的文件包含海量的元数据（可视化的坐标、版本号、依赖包信息）。
    *   **生成困难**：让 LLM 从零写一个合法的 UiPath .xaml 文件极其困难，极易出现 Schema 错误导致 Studio 无法打开。通常只能让 LLM 分析局部逻辑（如变量赋值逻辑），无法接管整个文件。
### 3. Automation Anywhere (AA) & Blue Prism
*   **文件格式**：**云端黑盒 / 专有 XML**。
*   **LLM 读写能力**：⭐ (极低)
*   **深度解析**：
    *   **AA**：基于 Web 的云原生架构，流程以 JSON/XML 存储在数据库中，用户很难接触到“源码文件”，LLM 完全无法介入文件层面的修改。
    *   **BP**：XML 文件主要用于导入导出，结构极其僵化，包含了大量图形界面的坐标信息，LLM 即使读懂了逻辑，也无法生成一个能被 BP 导入成功的有效文件。

## 二、 维度二：国内电商与应用生态适配
**痛点**：国内电商后台（千牛、抖店、拼多多）和办公软件（钉钉、飞书、企业微信）更新极快，且有大量反爬虫机制和非标准 UI 控件。
**结论**：**影刀在这一领域具有压倒性优势，海外 RPA 存在严重“水土不服”。**
### 1. 影刀 RPA
*   **适配程度**：⭐⭐⭐⭐⭐ (原生适配)
*   **实战优势**：
    *   **开箱即用**：影刀的应用市场（指令中心）内置了大量针对**淘宝、京东、抖音、小红书**的封装指令。例如“获取抖店订单”、“上传千牛发货”，无需自己去抓取网页元素。
    *   **对抗反爬与更新**：国内电商网页经常变动（改 Class 名、Shadow DOM 等），影刀团队更新指令库的速度极快。
    *   **特殊控件支持**：对国内特有的软件（如**WPS**、微信小程序、钉钉客户端）有专门的捕获驱动，捕获成功率远高于通用识别器。
### 2. UiPath / AA / Blue Prism
*   **适配程度**：⭐⭐ (水土不服)
*   **实战劣势**：
    *   **通用但不专精**：它们将“淘宝后台”视为一个普通的 Chrome 网页。你需要自己去写 Selector（选择器）来定位“发货按钮”。一旦淘宝改版，你的机器人就挂了，维护成本极高。
    *   **环境隔离问题**：国内许多银行网银助手、税务软件运行在极其封闭的沙盒或 IE 兼容模式下，海外 RPA 的标准浏览器插件经常失效，需要极深的技术配置才能跑通。
    *   **生态缺失**：你无法在 UiPath Marketplace 里找到高质量的“拼多多自动回复”现成组件，一切都要从零开发。
